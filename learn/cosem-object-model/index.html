The COSEM Object Model

The COSEM object model is the foundation of DLMS/COSEM communication. Understanding how objects work is essential for working with smart meters.

ℹ️

Everything in a DLMS/COSEM meter is an object - from simple values like voltage readings to complex structures like load profiles.

Object-Oriented Design

COSEM uses an object-oriented approach where:

  • Interface Classes define the structure (like templates or blueprints)
  • Objects are instances of these classes
  • Attributes hold the data values
  • Methods perform operations on the data

This design makes meters flexible, extensible, and standardized across vendors.

Interface Classes

Interface classes are predefined templates that specify what attributes and methods an object should have.

Common Interface Classes

1. Data (Class ID: 1)

  • Simplest class - holds a single value
  • Example: Meter serial number, software version
  • Attribute: value

2. Register (Class ID: 3)

  • Stores numerical values with units
  • Example: Energy consumption, voltage
  • Attributes: value, scaler_unit

3. Extended Register (Class ID: 4)

  • Register with capture time
  • Example: Maximum demand
  • Attributes: value, scaler_unit, capture_time

4. Demand Register (Class ID: 5)

  • Tracks maximum demand values
  • Attributes: current_average_value, last_average_value, status

5. Register Activation (Class ID: 6)

  • Manages multiple register values
  • Used for tariff switching

6. Profile Generic (Class ID: 7)

  • Most complex and powerful class
  • Stores time-series data (load profiles)
  • Attributes: buffer, capture_objects, capture_period, sort_method

7. Clock (Class ID: 8)

  • Manages meter time and date
  • Attributes: time, time_zone, daylight_savings
  • Methods: adjust_to_preset_time, shift_time

8. Script Table (Class ID: 9)

  • Stores executable scripts
  • Used for automation

9. Schedule (Class ID: 10)

  • Triggers scripts at specific times

10. Activity Calendar (Class ID: 20)

  • Manages tariff schedules
  • Defines active/passive tariff programs

11. Disconnect Control (Class ID: 70)

  • Remote connect/disconnect
  • Attributes: output_state, control_state
  • Methods: remote_disconnect, remote_reconnect

Object Structure

Every COSEM object has:

1. Logical Name (OBIS Code)

Unique identifier in format A.B.C.D.E.F

2. Class ID

Identifies which interface class the object belongs to

3. Attributes

Data values with specific indexes:

  • Attribute 0: Always reserved
  • Attribute 1: Usually logical_name
  • Attribute 2+: Class-specific attributes

4. Methods (Optional)

Operations that can be performed:

  • Method 0: Always reserved
  • Method 1+: Class-specific methods

Example: Register Object

Let's look at a concrete example - a Register object for total active energy:

{
  "class_id": 3,
  "logical_name": "1.0.1.8.0.255",
  "attributes": {
    "1": "1.0.1.8.0.255",        // logical_name
    "2": {                        // value
      "data": 12345678,
      "type": "double-long-unsigned"
    },
    "3": {                        // scaler_unit
      "scaler": -1,               // multiply by 10^-1
      "unit": 30                  // Wh (watt-hours)
    }
  }
}

Interpretation:

  • Value: 12345678 × 10⁻¹ = 1,234,567.8 Wh = 1,234.57 kWh

Example: Profile Generic Object

Load profile capturing energy consumption every 15 minutes:

{
  "class_id": 7,
  "logical_name": "1.0.99.1.0.255",
  "attributes": {
    "2": [                         // buffer (stored data)
      {
        "clock": "2025-10-02T00:00:00",
        "energy": 1234500,
        "voltage": 2300
      },
      {
        "clock": "2025-10-02T00:15:00",
        "energy": 1235000,
        "voltage": 2310
      }
    ],
    "3": [                         // capture_objects
      {
        "class_id": 8,
        "logical_name": "0.0.1.0.0.255",
        "attribute_index": 2       // clock value
      },
      {
        "class_id": 3,
        "logical_name": "1.0.1.8.0.255",
        "attribute_index": 2       // energy value
      },
      {
        "class_id": 3,
        "logical_name": "1.0.32.7.0.255",
        "attribute_index": 2       // voltage value
      }
    ],
    "4": 900,                      // capture_period (seconds)
    "5": 1                         // sort_method (FIFO)
  }
}
💡

The capture_objects attribute defines which objects to capture. Each entry specifies a class_id, OBIS code, and which attribute to read.

Accessing Objects

To read or write data, you need to know:

  1. OBIS Code - Where the object is
  2. Class ID - What type of object it is
  3. Attribute Index - Which piece of data you want

GET Request Example

Reading total active energy (Register object):

GET.request {
  invoke-id: 1,
  object: {
    class-id: 3,
    logical-name: "1.0.1.8.0.255",
    attribute-id: 2          // value attribute
  }
}

SET Request Example

Setting the meter clock:

SET.request {
  invoke-id: 2,
  object: {
    class-id: 8,
    logical-name: "0.0.1.0.0.255",
    attribute-id: 2,         // time attribute
    value: "2025-10-02T14:30:00"
  }
}

ACTION Request Example

Disconnecting power remotely:

ACTION.request {
  invoke-id: 3,
  object: {
    class-id: 70,
    logical-name: "0.0.96.3.10.255",
    method-id: 1             // remote_disconnect
  }
}

Object Relationships

Objects can reference other objects:

  • Profile Generic captures values from Register objects
  • Activity Calendar triggers Script Table objects
  • Schedule executes Script Table at specific times

This creates a powerful, interconnected data model.

Data Types

COSEM supports various data types:

Primitive Types

  • boolean
  • integer, long, unsigned
  • octet-string (bytes)
  • visible-string (text)
  • bcd (binary-coded decimal)
  • date, time, date-time

Structured Types

  • array - Ordered list of values
  • structure - Multiple fields of different types
  • compact-array - Optimized array

Example Structure

{
  "type": "structure",
  "value": [
    { "type": "double-long-unsigned", "value": 12345 },
    { "type": "integer", "value": -1 },
    { "type": "enum", "value": 30 }
  ]
}

This represents a register's value (12345), scaler (-1), and unit (30 = Wh).

Logical Devices

A physical meter can contain multiple logical devices:

  • Each logical device is an independent server
  • Has its own set of objects
  • Accessed via different associations

Example: A multi-utility meter might have:

  • Logical Device 1: Electricity metering
  • Logical Device 2: Gas metering
  • Logical Device 3: Water metering

Object Discovery

To explore what objects a meter has:

  1. Read Association Object (0.0.40.0.0.255)

    • Lists all accessible objects
    • Shows class IDs and OBIS codes

  2. Read Individual Objects

    • Get detailed information
    • Check available attributes and methods
⚠️

Not all objects are accessible to all users. Access rights depend on authentication level and association.

Best Practices

1. Know Your Interface Classes

Study the Blue Book specifications for each class you'll use.

2. Use Correct Data Types

Match the expected data type when setting values.

3. Check Access Rights

Some objects are read-only, others require high authentication.

4. Handle Errors Gracefully

Objects may return errors (e.g., "read-write denied", "object unavailable").

5. Batch Operations

Use GET-WITH-LIST to read multiple attributes in one request.

Real-World Usage

Reading Instantaneous Values

GET Register objects:
- 1.0.32.7.0.255 (Voltage L1)
- 1.0.52.7.0.255 (Voltage L2)
- 1.0.72.7.0.255 (Voltage L3)
- 1.0.1.7.0.255  (Active Power)

Retrieving Load Profile

GET Profile Generic object:
- 1.0.99.1.0.255 (Load profile)
  - Use selective access to get specific date range
  - Buffer can be large (thousands of entries)

Remote Disconnect

ACTION Disconnect Control object:
- 0.0.96.3.10.255
  - Method 1: remote_disconnect
  - Method 2: remote_reconnect

Summary

The COSEM object model provides:

  • Standardized structure across vendors
  • Flexible design for any metering application
  • Rich data types for complex information
  • Methods for remote operations
  • Relationships between objects

Understanding objects is key to mastering DLMS/COSEM!
