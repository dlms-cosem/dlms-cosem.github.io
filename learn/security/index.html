Security in DLMS/COSEM

Security is critical in smart metering. DLMS/COSEM includes comprehensive security features to protect against unauthorized access, data tampering, and cyber attacks.

⚠️

Smart meters control critical infrastructure and handle billing data. Proper security implementation is not optional!

Why Security Matters

Threats to Smart Meters

1. Unauthorized Access

  • Reading sensitive consumption data
  • Stealing personal information
  • Accessing billing records

2. Data Manipulation

  • Tampering with consumption readings
  • Altering tariff settings
  • Modifying billing periods

3. Service Disruption

  • Remote disconnection attacks
  • Denial of service
  • Firmware corruption

4. Infrastructure Attacks

  • Mass meter compromise
  • Grid instability
  • Cascading failures

Consequences

  • Financial losses (energy theft, billing errors)
  • Privacy violations (consumption profiling)
  • Safety risks (unauthorized disconnects)
  • Regulatory penalties (compliance failures)
  • Reputation damage (customer trust loss)

DLMS/COSEM Security Model

DLMS/COSEM uses a layered security approach:

┌────────────────────────────────────┐
│    Application Layer Security      │  <- Authentication & Authorization
├────────────────────────────────────┤
│    Data Layer Security             │  <- Encryption & Signing
├────────────────────────────────────┤
│    Transport Layer Security        │  <- TLS/VPN
└────────────────────────────────────┘

Security Levels

DLMS/COSEM defines multiple authentication levels:

Level 0: No Security

  • No authentication
  • No encryption
  • Never use in production!
  • Only for testing/development

Level 1: Low Level Security (LLS)

  • Password-based authentication
  • Simple shared secret
  • No encryption
  • Legacy systems only

Example:

Password: "MyMeterPass"
⚠️

LLS provides minimal security. Use HLS whenever possible.

Level 2-7: High Level Security (HLS)

  • Challenge-response authentication
  • Strong cryptography
  • Mutual authentication
  • Supports encryption

HLS Variants:

  • HLS-GMAC: Authentication only
  • HLS-GMAC with broadcast key: Group authentication
  • HLS-SHA-256: SHA-256 based
  • HLS-ECDSA: Elliptic curve signatures

Authentication Process

HLS Challenge-Response

Step 1: Association Request (AARQ)

Client -> Server: I want to connect with HLS authentication
                  My challenge: [random_data_1]

Step 2: Association Response (AARE)

Server -> Client: Connection accepted
                  Your challenge result: [f(random_data_1, secret)]
                  My challenge: [random_data_2]

Step 3: Client Verification

Client:
  - Verify server's challenge response
  - Calculate response to server's challenge
  - Send: [f(random_data_2, secret)]

Step 4: Server Verification

Server:
  - Verify client's challenge response
  - If valid: Grant access
  - If invalid: Reject connection

Python Example (Simplified)

import hmac
import hashlib
import os

def hls_authenticate(challenge, secret):
    """Compute HLS-GMAC response"""
    return hmac.new(
        secret.encode(),
        challenge,
        hashlib.sha256
    ).digest()

# Client generates challenge
client_challenge = os.urandom(16)

# Server responds
server_challenge = os.urandom(16)
server_response = hls_authenticate(client_challenge, "shared_secret")

# Client verifies and responds
client_response = hls_authenticate(server_challenge, "shared_secret")

Encryption

AES-GCM (Galois/Counter Mode)

DLMS/COSEM uses AES-128-GCM for encryption:

  • Encryption: Protects data confidentiality
  • Authentication: Verifies data integrity
  • Nonce: Prevents replay attacks

Properties:

  • Key size: 128 bits (16 bytes)
  • Block size: 128 bits
  • Tag size: 96 or 128 bits

Encryption Keys

Three types of keys:

  1. Authentication Key (AK)

    • Used for authentication
    • Shared secret

  2. Global Encryption Key (GEK)

    • Encrypts unicast communication
    • Unique per meter

  3. Broadcast Encryption Key (BEK)

    • Encrypts broadcast messages
    • Shared across meters
ℹ️

Keys should be rotated periodically and never transmitted in plain text!

Security Suites

DLMS/COSEM defines security suites combining algorithms:

Suite 0: AES-GCM-128

  • Encryption: AES-128-GCM
  • Authentication: GMAC
  • Key agreement: None

Suite 1: ECDH-ECDSAAES-GCM-128

  • Encryption: AES-128-GCM
  • Authentication: ECDSA
  • Key agreement: ECDH (Elliptic Curve Diffie-Hellman)

Suite 2: ECDH-ECDSAAES-GCM-256

  • Encryption: AES-256-GCM
  • Authentication: ECDSA
  • Key agreement: ECDH

Security Context

Each DLMS message can have a security context:

{
  securityControl: 0x30,     // Authentication + Encryption
  invocationCounter: 12345,  // Replay protection
  securityHeader: {
    key: "GEK",              // Which key to use
    transactionId: 0x0001
  },
  encryptedData: [...],      // Encrypted APDU
  authenticationTag: [...]   // GMAC tag
}

Security Control Byte

Bits define security options:

Bit 0-2: Security suite (0-7)
Bit 3:   Authenticated (0=No, 1=Yes)
Bit 4:   Encrypted (0=No, 1=Yes)
Bit 5:   Broadcast key (0=No, 1=Yes)
Bit 6:   Compressed (0=No, 1=Yes)
Bit 7:   Reserved

Examples:

  • 0x00: No security
  • 0x10: Authentication only
  • 0x30: Authentication + Encryption

Access Control

DLMS/COSEM implements fine-grained access control:

Association Types

1. Public Client

  • No authentication
  • Read-only access
  • Limited objects

2. Management Client

  • HLS authentication
  • Read-write access
  • Configuration allowed

3. Firmware Update Client

  • Strong authentication
  • Firmware image transfer
  • Critical operations

Access Rights

Each object attribute/method has access rights:

{
  objectAccess: {
    "1.0.1.8.0.255": {  // Total energy
      read: ["public", "management"],
      write: []           // Nobody can write
    },
    "0.0.1.0.0.255": {  // Clock
      read: ["public", "management"],
      write: ["management"]  // Only management
    },
    "0.0.96.3.10.255": {  // Disconnect control
      read: ["management"],
      write: [],
      action: ["management"]  // Remote disconnect
    }
  }
}
💡

Follow the principle of least privilege - grant minimum necessary access!

Key Management

Key Distribution

Options:

  1. Pre-installation: Keys loaded during manufacturing
  2. Physical installation: Keys loaded via optical port
  3. Key transport: Encrypted key delivery
  4. Key agreement: ECDH key derivation

Key Hierarchy

Master Key (MK)
  ├─ Authentication Key (AK)
  ├─ Global Encryption Key (GEK)
  └─ Broadcast Encryption Key (BEK)

Master key is stored securely (HSM, secure element).

Key Update

Process:

  1. Authenticate with current key
  2. Send new key (encrypted with current key)
  3. Activate new key
  4. Verify with new key
  5. Delete old key
def update_key(old_key, new_key):
    # Authenticate with old key
    authenticate(old_key)

    # Encrypt new key with old key
    encrypted_new_key = encrypt(new_key, old_key)

    # Send to meter
    send_key_update(encrypted_new_key)

    # Switch to new key
    current_key = new_key

Firmware Security

Secure Firmware Update

Steps:

  1. Verification: Check signature
  2. Integrity: Validate checksum
  3. Authentication: Verify source
  4. Encryption: Decrypt image
  5. Installation: Flash firmware
  6. Validation: Boot and verify

Image Transfer Object (Class ID: 18)

{
  classId: 18,
  logicalName: "0.0.44.0.0.255",
  attributes: {
    imageBlockSize: 512,
    imageTransferredBlocksStatus: "...",
    imageFirstNotTransferredBlockNumber: 0,
    imageTransferEnabled: true,
    imageTransferStatus: 0  // Not initiated
  },
  methods: {
    imageTransferInitiate: 1,
    imageBlockTransfer: 2,
    imageVerify: 3,
    imageActivate: 4
  }
}
⚠️

Always verify firmware signatures before installation!

Certificate Management

X.509 Certificates

For Suite 1 and Suite 2:

Certificate:
  Serial Number: 0x12345678
  Issuer: CN=Utility CA, O=Power Company
  Subject: CN=Meter-001, O=Power Company
  Public Key: [ECDSA P-256 public key]
  Signature: [CA signature]

Certificate Lifecycle

  1. Enrollment: Generate key pair, get certificate
  2. Validation: Check certificate chain, revocation
  3. Usage: Authenticate, sign, encrypt
  4. Renewal: Before expiration
  5. Revocation: If compromised

Best Practices

1. Use Strong Authentication

  • ✅ HLS-GMAC or better
  • ❌ Never use LLS in production
  • ❌ Never use no authentication

2. Enable Encryption

  • ✅ AES-GCM for all sensitive data
  • ✅ Encrypt firmware updates
  • ✅ Use authenticated encryption

3. Manage Keys Securely

  • ✅ Rotate keys regularly
  • ✅ Use secure key storage (HSM)
  • ✅ Implement key backup/recovery
  • ❌ Never hardcode keys in software

4. Implement Access Control

  • ✅ Principle of least privilege
  • ✅ Separate client roles
  • ✅ Audit access attempts
  • ❌ Don't use shared accounts

5. Monitor and Log

  • ✅ Log all authentication attempts
  • ✅ Alert on suspicious activity
  • ✅ Regular security audits
  • ✅ Incident response plan

6. Network Security

  • ✅ Use VPN for remote access
  • ✅ Firewall rules
  • ✅ Network segmentation
  • ❌ Don't expose meters to public internet

7. Physical Security

  • ✅ Tamper detection
  • ✅ Secure meter housing
  • ✅ Optical port protection
  • ✅ Anti-theft measures

Common Vulnerabilities

1. Weak Passwords

Problem: Default or simple passwords Solution: Strong, unique passwords + HLS

2. Replay Attacks

Problem: Reusing captured messages Solution: Invocation counters, timestamps

3. Man-in-the-Middle

Problem: Attacker intercepts communication Solution: Mutual authentication, encryption

4. Key Extraction

Problem: Keys extracted from meter Solution: Secure elements, tamper detection

5. Firmware Manipulation

Problem: Malicious firmware installation Solution: Signed firmware, secure boot

Security Checklist

Before deployment:

  • Strong authentication enabled (HLS)
  • Encryption enabled (AES-GCM)
  • Unique keys per meter
  • Key management process defined
  • Access control configured
  • Audit logging enabled
  • Network security implemented
  • Firmware update process secured
  • Tamper detection active
  • Incident response plan ready
  • Security training completed
  • Compliance verified

Summary

DLMS/COSEM security provides:

  • Authentication: Verify identity (HLS)
  • Encryption: Protect data (AES-GCM)
  • Integrity: Detect tampering (GMAC)
  • Access Control: Manage permissions
  • Audit: Track security events
  • Key Management: Secure key lifecycle

Proper security implementation protects meters, customers, and infrastructure!
