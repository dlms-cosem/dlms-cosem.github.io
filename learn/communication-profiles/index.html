Communication Profiles

DLMS/COSEM is transport-agnostic, meaning it can work over various physical and network layers. Communication profiles define how DLMS messages are transmitted over different media.

The same DLMS/COSEM application layer can work over serial, TCP/IP, wireless, or power line networks - just by changing the communication profile!

Common Profiles

1. HDLC Profile (IEC 62056-46)

High-Level Data Link Control - The most common profile for serial connections.

Characteristics:

  • Point-to-point or multi-drop
  • Frame-based communication
  • Built-in error detection (CRC)
  • Flow control
  • Supports addressing

Physical Layers:

  • RS-232 (DB9, point-to-point)
  • RS-485 (multi-drop, up to 32 devices)
  • Optical probe (IEC 62056-21)
  • M-Bus

  • Direct meter connection (optical, serial)

2. TCP/IP Profile (IEC 62056-47)

Wrapper profile - Encapsulates DLMS in TCP packets.

Characteristics:

  • Network-based communication
  • Client-server model
  • Standard TCP/IP stack
  • Port 4059 (default)

Connection Types:

  • LAN (Ethernet)
  • WAN (Internet)
  • GPRS/LTE (mobile networks)
  • Wi-Fi

When to Use:

  • Network-enabled meters
  • Remote access over IP
  • Cloud-based systems
  • High-bandwidth applications

3. SMS Profile (IEC 62056-48)

Short Message Service - For GSM/cellular meters.

Characteristics:

  • Works with any GSM meter
  • Limited message size (160 bytes)
  • Lower cost than data connection
  • Suitable for infrequent reads

Message Format:

  • Binary or text mode
  • Compressed DLMS data
  • Supports fragmentation for large messages

When to Use:

  • Remote rural locations
  • Minimal data transmission
  • Fallback communication
  • Low-cost deployments

4. M-Bus Profile (IEC 62056-8-4)

Meter-Bus integration profile.

Characteristics:

  • Connects DLMS meters to M-Bus systems
  • Two-wire bus connection
  • Master-slave architecture
  • Low power consumption

When to Use:

  • European metering infrastructure
  • Multi-utility buildings
  • Sub-metering applications
  • M-Bus concentrators

5. PLC Profile (IEC 62056-8-5)

Power Line Communication - Uses electrical wiring.

Characteristics:

  • Communication over power lines
  • No additional wiring needed
  • Various PLC standards (G3-PLC, PRIME, etc.)
  • Suitable for LV networks

When to Use:

  • Dense urban areas
  • Existing power infrastructure
  • Smart grid applications
  • Neighborhood networks

6. Wireless M-Bus (IEC 62056-8-10)

Radio frequency meter reading.

Characteristics:

  • 868 MHz (Europe) / 915 MHz (US)
  • Walk-by/drive-by reading
  • Low power consumption
  • Uni-directional or bi-directional

When to Use:

  • Water/gas/heat meters
  • Mobile data collection
  • Battery-powered meters
  • Multi-family buildings

HDLC in Detail

Connection Setup

  1. SNRM (Set Normal Response Mode)

    • Client initiates connection
    • Negotiates parameters

  2. UA (Unnumbered Acknowledgment)

    • Server accepts connection
    • Confirms parameters

  3. Data Exchange

    • Information frames (I-frames)
    • Acknowledgments

  4. DISC (Disconnect)

    • Clean connection termination

HDLC Frame Types

Information Frame (I-frame):

  • Carries DLMS data
  • Sequence numbers
  • Requires acknowledgment

Supervisory Frame (S-frame):

  • RR (Receive Ready)
  • RNR (Receive Not Ready)
  • REJ (Reject)

Unnumbered Frame (U-frame):

  • SNRM (connection setup)
  • UA (acknowledgment)
  • DISC (disconnect)
  • DM (disconnect mode)

Example HDLC Communication

Client -> Server: SNRM (Setup connection)
Server -> Client: UA (Connection accepted)

Client -> Server: I-frame with GET.request
Server -> Client: I-frame with GET.response

Client -> Server: RR (Receive Ready)
Server -> Client: I-frame with more data

Client -> Server: DISC (Disconnect)
Server -> Client: UA (Disconnected)

TCP/IP in Detail

Connection Process

  1. TCP Connection

    Client -> Server: TCP SYN
Server -> Client: TCP SYN-ACK
Client -> Server: TCP ACK

  2. DLMS Association

    Client -> Server: AARQ (Association Request)
Server -> Client: AARE (Association Response)

  3. Data Exchange

    Client -> Server: GET/SET/ACTION requests
Server -> Client: Responses

  4. Disconnection

    Client -> Server: RLRQ (Release Request)
Server -> Client: RLRE (Release Response)
Client -> Server: TCP FIN

TCP Wrapper Format

// TCP Wrapper packet
{
  version: 0x01,              // Wrapper version
  length: 0x0012,             // Payload length (18 bytes)
  payload: [0x60, 0x10, ...]  // DLMS APDU
}

Python Example

import socket

# Connect to meter
sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
sock.connect(('192.168.1.100', 4059))

# Send AARQ (Association Request)
aarq = bytes([0x00, 0x01, 0x00, 0x12, 0x60, 0x10, ...])
sock.send(aarq)

# Receive AARE (Association Response)
aare = sock.recv(1024)

# Send GET request
get_request = bytes([0x00, 0x01, 0x00, 0x08, 0xC0, 0x01, ...])
sock.send(get_request)

# Receive response
response = sock.recv(1024)

# Close
sock.close()

Choosing the Right Profile

1. Installation Environment

2. Data Requirements

  • Real-time data? â†’ TCP/IP
  • Daily reads? â†’ SMS or PLC
  • On-demand access? â†’ HDLC

3. Cost Constraints

  • Low budget? â†’ HDLC or SMS
  • Medium budget? â†’ PLC
  • Higher budget? â†’ TCP/IP with cellular

4. Security Needs

  • High security? â†’ TCP/IP with VPN

  • Basic security? â†’ HDLC with authentication

5. Scalability

  • Few meters? â†’ HDLC
  • Hundreds? â†’ PLC or TCP/IP

Tier 1: Field Communication

Profile Configuration

HDLC Parameters

{
  baudRate: 9600,           // Or 19200, 38400, 115200
  dataBits: 8,
  stopBits: 1,
  parity: 'none',
  maxFrameSize: 128,        // Negotiate with server
  windowSize: 1,            // Number of unacknowledged frames
  interCharTimeout: 30,     // Milliseconds
  responseTimeout: 3000     // Milliseconds

TCP/IP Parameters

{
  host: '192.168.1.100',
  port: 4059,
  timeout: 30000,           // Connection timeout
  keepAlive: true,
  maxPacketSize: 65535
}

Common Issues

HDLC Problems

Frame Errors:

  • Bad CRC â†’ Check cable quality
  • Timeout â†’ Adjust response timeout
  • Wrong addressing â†’ Verify meter address

Solutions:

  • Use shielded cables
  • Check baud rate settings
  • Verify physical connections

TCP/IP Problems

Connection Refused:

  • Firewall blocking
  • Wrong IP address
  • Meter not listening

Solutions:

  • Check firewall rules
  • Verify network configuration
  • Confirm port 4059 is open

Packet Loss:

  • Network congestion
  • Poor signal (cellular)

Solutions:

  • Implement retries
  • Use QoS settings
  • Optimize packet size

Best Practices

1. Use Standard Ports

  • TCP: Port 4059
  • UDP: Port 4059
  • Avoid custom ports unless necessary

2. Implement Timeouts

  • Connection timeout: 30s
  • Read timeout: 10s
  • Retry failed operations

3. Handle Errors Gracefully

  • Log communication errors
  • Implement retry logic
  • Fall back to alternate profile

4. Optimize Bandwidth

  • Batch multiple requests
  • Compress data when possible
  • Use appropriate profile for data volume

5. Secure Communication

  • Use encryption (TLS for TCP)
  • Authenticate devices
  • Implement access control

Summary

Communication profiles provide:

  • Flexibility - Work over any medium
  • Standardization - Consistent behavior
  • Scalability - From single meter to millions
  • Reliability - Error detection and correction
  • Choice - Select best fit for your needs

Understanding profiles is key to deploying DLMS/COSEM systems!
